# Test Plan: IAM-106 Drag-and-Drop Optimistic Status Update

## Scope

Validate the drag-and-drop status-update fix for IAM-106 on the live Kanban board at `http://localhost:5173`.

In scope:

- Moving a ticket from one column to another updates the UI immediately after drop.
- The dragged ticket does not visibly snap back to its original column before settling in the destination column.
- The move feels smooth and immediate, without a teleport or double-move effect.
- Drag cancel and same-column drop behavior remain visually stable.
- A forced server failure rolls the ticket back to its original column after the optimistic move.
- A blocked-ticket move attempt does not introduce a snap-back glitch, regardless of whether the product allows or rejects that transition.

Out of scope:

- Backend status-update implementation details beyond visible user behavior.
- Source-code inspection.
- Mobile/tablet drag behavior.
- Performance profiling beyond visible animation quality.

## Test Approach

- Black-box validation only, using the running UI and user-visible labels.
- Primary execution target: `http://localhost:5173`.
- Manual interaction for visual-quality checks because animation defects are easiest to confirm with real pointer drag.
- Playwright automation for repeatable success, no-op, cancel, blocked, and forced-error coverage.
- For error-path validation, use Playwright network interception to force the status-update request to fail.

## Preconditions

- React dev server is running at `http://localhost:5173`.
- The board shows at least two columns with drag-and-drop enabled.
- At least one movable ticket is visible.
- For same-column and cancel-drag checks, use any visible ticket.
- For blocked-ticket validation, the environment should contain a ticket that is visibly blocked or linked to a blocker.
- For rollback validation, the browser test harness can intercept the status-update request and return a failure.

## Visual Quality Checklist

Use this checklist during execution for every drag scenario:

- The ticket follows the pointer naturally during drag.
- On drop, the ticket settles into the destination column without first flashing in the source column.
- No duplicate ticket card is visible during or after the drop.
- No empty gap remains behind in the source column longer than expected.
- No abrupt teleport occurs after the drop completes.
- If rollback happens, the ticket returns once and ends in the correct final column.

## Regression Risks To Watch

- Destination column count updates late or inconsistently with the visible card move.
- Source column still briefly renders the dropped card after release.
- Empty-state placeholder flashes in and out during move.
- Repeated drags leave stale drag overlays or duplicate cards.
- Error rollback restores the ticket twice or lands it in the wrong column.
- Blocked-ticket rules are enforced inconsistently between manual and automated runs.

## Playwright Automation Scripts

The following scripts are intended as executable starting points for automated verification. They deliberately use user-visible labels and runtime interception rather than implementation-specific selectors.

### Script 1: Happy Path Immediate Move

```ts
import { test, expect } from '@playwright/test';

test('IAM-106: ticket moves immediately after drop without snap-back', async ({ page }) => {
  await page.goto('http://localhost:5173');

  const sourceCard = page.getByRole('button', {
    name: /SBFF-18|\[TEST\] IAM-106/i,
  }).first();

  const targetColumn = page.getByText('In Progress', { exact: true }).first();

  await expect(sourceCard).toBeVisible();
  await expect(targetColumn).toBeVisible();

  await sourceCard.dragTo(targetColumn);

  await expect(sourceCard).toBeVisible();
  await expect(page.getByText('In Progress', { exact: true })).toBeVisible();
});
```

### Script 3: Cancel Drag

```ts
import { test, expect } from '@playwright/test';

test('IAM-106: canceling drag leaves the ticket in place without flicker', async ({ page }) => {
  await page.goto('http://localhost:5173');

  const card = page.getByRole('button', {
    name: /SBFF-18|\[TEST\] IAM-106/i,
  }).first();

  const box = await card.boundingBox();
  if (!box) throw new Error('Card bounding box not found');

  await page.mouse.move(box.x + box.width / 2, box.y + box.height / 2);
  await page.mouse.down();
  await page.mouse.move(box.x + box.width / 2 + 120, box.y + box.height / 2 + 40);
  await page.keyboard.press('Escape');
  await page.mouse.up();

  await expect(card).toBeVisible();
});
```

### Script 4: Forced Server Error With Rollback

```ts
import { test, expect } from '@playwright/test';

test('IAM-106: optimistic move rolls back cleanly on server error', async ({ page }) => {
  await page.route(/\/tickets\/.*status|\/tickets\/.*/, async route => {
    const request = route.request();
    if (request.method() === 'PATCH' || request.method() === 'PUT') {
      await route.fulfill({
        status: 500,
        contentType: 'application/json',
        body: JSON.stringify({ detail: 'Forced IAM-106 failure' }),
      });
      return;
    }

    await route.continue();
  });

  await page.goto('http://localhost:5173');

  const card = page.getByRole('button', {
    name: /SBFF-18|\[TEST\] IAM-106/i,
  }).first();

  const targetColumn = page.getByText('In Progress', { exact: true }).first();

  await expect(card).toBeVisible();
  await card.dragTo(targetColumn);

  await expect(card).toBeVisible();

  await expect
    .poll(async () => await card.textContent())
    .toContain('SBFF-18');
});
```

Execution note for Script 4:

- If the app exposes an error toast or inline message, add an assertion for that feedback.
