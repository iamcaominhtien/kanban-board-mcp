# Test Plan: IAM-88 and IAM-89 Markdown and Image Support

## Scope

This plan validates two feature areas that share the same Markdown editing and image-upload implementation:

- IAM-88: Markdown and image support for Work Log notes
- IAM-89: Markdown and image support for the Test Case Proof field

Coverage includes:

- Markdown authoring for bold, italic, lists, links, headings, quotes, and code blocks
- Markdown rendering in read/view mode through the shared renderer
- Clipboard image paste into the shared editor
- File-picker image upload from the shared editor toolbar
- Supported image formats: `png`, `jpg`, `jpeg`, `gif`, `webp`
- Persistence after save, collapse/expand, modal close, and page reload
- Error-handling and boundary behavior relevant to shared editor logic

Out of scope:

- Drag-and-drop uploads
- Image resize, crop, compression, or delete workflows
- Backend storage cleanup after referenced images are removed from content
- Browser-specific clipboard quirks outside the primary QA browser

## Implementation Notes

Observed behavior from the implementation files under test:

- `MarkdownEditor` provides toolbar insertion for markdown syntax and an `Img` file-picker action
- Clipboard paste uploads only supported `image/png`, `image/jpeg`, `image/gif`, and `image/webp` MIME types
- Unsupported file types selected outside the native file-picker filter are ignored client-side
- Uploads insert temporary markdown placeholders and replace them with returned markdown on success
- Failed uploads remove the temporary placeholder
- `MarkdownRenderer` renders markdown through `react-markdown` with `remark-gfm` and a sanitized schema that allows safe `img` tags with `http` and `https` sources
- Work Log view mode renders saved notes through `MarkdownRenderer`
- Test Case Proof uses `MarkdownEditor` in edit mode and the same markdown rendering path in view mode

## Test Approach

- Primary execution method: Playwright-based UI testing against the local app
- Setup strategy: create isolated `[TEST]` tickets and test cases so saved markdown and images can be reloaded safely
- Assertion strategy:
  - Visual: verify formatted text and images in view mode
  - Behavioral: verify paste/upload, save, collapse/expand, and reload persistence
  - Negative: verify unsupported content does not corrupt saved data or leave upload placeholders behind

## Assumptions and Preconditions

- Frontend and backend are running in the same environment
- The image upload API used by `uploadDescriptionImage` is reachable
- The tester can create or edit at least one ticket with an expandable Work Log section and a Test Cases section
- Test image fixtures are available locally for all supported formats
- Reload means either a hard browser refresh or reopening the same ticket after data has been persisted

## Cross-Feature Regression Focus

These checks are especially important because IAM-88 and IAM-89 share the same editor and renderer:

- Paste and file-picker image flows behave the same in Work Log and Proof
- Supported image formats behave consistently in both surfaces
- Non-image paste remains normal text paste in both surfaces
- Failed or unsupported uploads do not leave placeholder markdown behind
- Markdown render output is consistent between Work Log viewer and Proof view mode
